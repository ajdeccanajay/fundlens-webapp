# Advanced RAG Techniques Explained

## 1. HyDE (Hypothetical Document Embeddings)

### What is HyDE?

HyDE is a retrieval technique that improves semantic search by generating a hypothetical answer to the query, then using that answer for retrieval instead of (or in addition to) the original query.

### The Problem HyDE Solves

**Traditional Semantic Search Problem:**
- User query: "Who are NVDA's competitors?"
- Query embedding: [0.2, 0.5, 0.1, ...]
- Document text: "NVIDIA faces competition from Advanced Micro Devices (AMD), Intel Corporation, and emerging AI chip startups..."
- Document embedding: [0.3, 0.4, 0.2, ...]

The query and document embeddings may not be similar because:
- Query is a question (short, interrogative)
- Document is a statement (long, declarative)
- Different vocabulary and structure

### How HyDE Works

**Step 1: Generate Hypothetical Answer**
```
Query: "Who are NVDA's competitors?"

Claude generates hypothetical answer:
"NVIDIA's main competitors include AMD, Intel, and AI chip startups. 
AMD competes in GPUs for gaming and data centers. Intel competes in 
data center AI chips. Startups like Cerebras and Graphcore are 
emerging competitors in specialized AI hardware."
```

**Step 2: Embed Hypothetical Answer**
```
Hypothetical answer embedding: [0.29, 0.41, 0.19, ...]
```

**Step 3: Retrieve Using Hypothetical Embedding**
```
The hypothetical answer embedding is much closer to the actual 
document embedding because both are declarative statements about 
competitors.

Result: Better retrieval accuracy!
```

### HyDE Implementation in Our System

```typescript
// In semantic-retriever.service.ts

async retrieveWithHyDE(query: string, filter: MetadataFilter): Promise<ChunkResult[]> {
  // Step 1: Generate hypothetical answer
  const hypotheticalAnswer = await this.bedrock.generateHypotheticalAnswer(query);
  
  // Step 2: Retrieve using both query and hypothetical answer
  const queryResults = await this.bedrock.retrieve(query, filter, 10);
  const hydeResults = await this.bedrock.retrieve(hypotheticalAnswer, filter, 10);
  
  // Step 3: Merge and deduplicate
  const merged = this.mergeAndDeduplicate(queryResults, hydeResults);
  
  // Step 4: Re-rank merged results
  const reranked = await this.reranker.rerank(query, merged);
  
  return reranked.slice(0, 5); // Top 5 results
}
```

### When to Use HyDE

✅ **Use HyDE when:**
- Query is a question that may not match document phrasing
- Looking for specific information (competitors, risks, policies)
- Initial retrieval returns low-confidence results

❌ **Don't use HyDE when:**
- Query is already similar to document text
- Looking for exact phrases or numbers
- Latency is critical (HyDE adds one LLM call)

### HyDE Example

**Query:** "What are NVDA's competitive advantages?"

**Hypothetical Answer (generated by Claude):**
```
NVIDIA's competitive advantages include:
1. Leading GPU technology with CUDA ecosystem
2. Strong brand recognition in gaming and AI markets
3. First-mover advantage in AI training hardware
4. Extensive software ecosystem and developer tools
5. Strategic partnerships with cloud providers
```

**Retrieval Result:**
This hypothetical answer will match documents that discuss NVIDIA's strengths, even if they don't use the exact phrase "competitive advantages."

## 2. Iterative Retrieval

### What is Iterative Retrieval?

Iterative retrieval is a technique where the system makes multiple retrieval passes, using information from earlier passes to refine later queries.

### The Problem Iterative Retrieval Solves

**Single-Pass Retrieval Problem:**
- Query: "Compare NVDA's and AMD's competitive positions"
- First retrieval: Gets NVDA's competitors
- Missing: AMD's specific competitive positioning
- Result: Incomplete answer

### How Iterative Retrieval Works

**Pass 1: Initial Retrieval**
```
Query: "Compare NVDA's and AMD's competitive positions"
Retrieved: NVDA competitor information
Confidence: 0.6 (low - missing AMD details)
```

**Pass 2: Gap Analysis**
```
System detects: "We have NVDA info but not AMD info"
Generated follow-up query: "What is AMD's competitive position in GPUs?"
Retrieved: AMD competitive positioning
Confidence: 0.85 (high)
```

**Pass 3: Synthesis**
```
Combine Pass 1 + Pass 2 results
Final confidence: 0.85
Complete answer with both companies
```

### Iterative Retrieval Implementation

```typescript
// In rag.service.ts

async queryWithIterativeRetrieval(query: string): Promise<RAGResponse> {
  let allNarratives: ChunkResult[] = [];
  let confidence = 0;
  let iteration = 0;
  const maxIterations = 2;
  
  while (confidence < 0.8 && iteration < maxIterations) {
    iteration++;
    
    // Retrieve for current query
    const result = await this.semanticRetriever.retrieve({
      query: iteration === 1 ? query : this.generateFollowUpQuery(query, allNarratives),
      tickers: this.extractTickers(query),
      numberOfResults: 5
    });
    
    // Add to accumulated results
    allNarratives = this.mergeAndDeduplicate(allNarratives, result.narratives);
    
    // Calculate confidence
    confidence = this.calculateConfidence(allNarratives, query);
    
    this.logger.log(`Iteration ${iteration}: ${result.narratives.length} chunks, confidence ${confidence}`);
    
    if (confidence >= 0.8) {
      this.logger.log(`Confidence threshold met, stopping iteration`);
      break;
    }
  }
  
  // Generate response from all accumulated narratives
  return this.generateResponse(query, allNarratives);
}

private generateFollowUpQuery(originalQuery: string, existingNarratives: ChunkResult[]): string {
  // Analyze what's missing
  const gaps = this.identifyGaps(originalQuery, existingNarratives);
  
  // Generate targeted follow-up query
  return `Provide more details about: ${gaps.join(', ')}`;
}
```

### Iterative Retrieval Example

**Original Query:** "What are NVDA's growth drivers and risks?"

**Iteration 1:**
```
Retrieved: 3 chunks about growth drivers (data center demand, AI adoption)
Confidence: 0.65 (missing risk information)
Gap identified: "risks"
```

**Iteration 2:**
```
Follow-up query: "What are NVDA's business risks?"
Retrieved: 4 chunks about risks (supply chain, competition, regulation)
Confidence: 0.90 (complete information)
```

**Final Response:**
```
Growth Drivers:
- Data center demand for AI training
- Gaming GPU market expansion
- Automotive AI partnerships

Risks:
- Supply chain dependencies on TSMC
- Increasing competition from AMD and Intel
- Regulatory scrutiny in China market
```

### When to Use Iterative Retrieval

✅ **Use iterative retrieval when:**
- Initial retrieval has low confidence (<0.7)
- Query has multiple parts (growth drivers AND risks)
- Missing information can be identified programmatically

❌ **Don't use iterative retrieval when:**
- Initial retrieval has high confidence (>0.8)
- Latency is critical (each iteration adds retrieval time)
- Query is simple and focused

## 3. Prompt Fine-Tuning Infrastructure

### What is Prompt Fine-Tuning Infrastructure?

A system for managing, versioning, testing, and optimizing prompts used for extraction tasks, without requiring code changes.

### The Problem It Solves

**Current State (Hardcoded Prompts):**
```typescript
// In bedrock.service.ts
const prompt = `Extract competitor names from the following text: ${text}`;
```

**Problems:**
- Can't update prompts without code deployment
- No way to test prompt variants
- No tracking of which prompts work best
- Can't adapt to new extraction patterns

### Prompt Fine-Tuning Infrastructure Components

#### Component 1: Prompt Library (Database)

```sql
CREATE TABLE prompt_templates (
  id SERIAL PRIMARY KEY,
  name VARCHAR(255) NOT NULL,
  version INTEGER NOT NULL,
  intent_type VARCHAR(50) NOT NULL, -- 'competitive_intelligence', 'mda_intelligence', etc.
  template TEXT NOT NULL,
  few_shot_examples JSONB,
  created_at TIMESTAMP DEFAULT NOW(),
  is_active BOOLEAN DEFAULT false,
  performance_score DECIMAL(3,2), -- 0.00 to 1.00
  usage_count INTEGER DEFAULT 0,
  success_count INTEGER DEFAULT 0,
  UNIQUE(name, version)
);
```

#### Component 2: Prompt Service

```typescript
// src/rag/prompt.service.ts

@Injectable()
export class PromptService {
  constructor(private readonly prisma: PrismaService) {}
  
  /**
   * Get the best-performing active prompt for an intent type
   */
  async getPrompt(intentType: string): Promise<PromptTemplate> {
    return this.prisma.promptTemplate.findFirst({
      where: {
        intentType,
        isActive: true
      },
      orderBy: {
        performanceScore: 'desc' // Best performing first
      }
    });
  }
  
  /**
   * Render prompt with variables and few-shot examples
   */
  renderPrompt(template: PromptTemplate, variables: Record<string, any>): string {
    let prompt = template.template;
    
    // Replace variables
    for (const [key, value] of Object.entries(variables)) {
      prompt = prompt.replace(`{{${key}}}`, value);
    }
    
    // Add few-shot examples
    if (template.fewShotExamples && template.fewShotExamples.length > 0) {
      const examples = template.fewShotExamples
        .map(ex => `Example:\nInput: ${ex.input}\nOutput: ${JSON.stringify(ex.output, null, 2)}`)
        .join('\n\n');
      
      prompt = `${examples}\n\n${prompt}`;
    }
    
    return prompt;
  }
  
  /**
   * Record prompt usage and success
   */
  async recordUsage(promptId: number, success: boolean, confidence: number) {
    await this.prisma.promptTemplate.update({
      where: { id: promptId },
      data: {
        usageCount: { increment: 1 },
        successCount: success ? { increment: 1 } : undefined,
        performanceScore: this.calculatePerformanceScore(confidence)
      }
    });
  }
  
  private calculatePerformanceScore(confidence: number): number {
    // Exponential moving average of confidence scores
    return confidence;
  }
}
```

#### Component 3: A/B Testing

```typescript
// src/rag/prompt-ab-test.service.ts

@Injectable()
export class PromptABTestService {
  /**
   * Get prompt variant for A/B testing
   * Randomly selects between active variants based on traffic allocation
   */
  async getPromptVariant(intentType: string, userId: string): Promise<PromptTemplate> {
    const variants = await this.prisma.promptTemplate.findMany({
      where: {
        intentType,
        isActive: true
      }
    });
    
    if (variants.length === 1) {
      return variants[0];
    }
    
    // Consistent hashing for user-based variant assignment
    const hash = this.hashUserId(userId);
    const variantIndex = hash % variants.length;
    
    return variants[variantIndex];
  }
  
  /**
   * Compare performance of prompt variants
   */
  async compareVariants(intentType: string): Promise<VariantComparison> {
    const variants = await this.prisma.promptTemplate.findMany({
      where: { intentType, isActive: true }
    });
    
    return {
      variants: variants.map(v => ({
        version: v.version,
        usageCount: v.usageCount,
        successRate: v.successCount / v.usageCount,
        performanceScore: v.performanceScore
      })),
      winner: variants.reduce((best, current) => 
        current.performanceScore > best.performanceScore ? current : best
      )
    };
  }
}
```

### Prompt Fine-Tuning Workflow

**Week 1: Initial Deployment**
```
1. Deploy prompt v1 for competitive intelligence
2. Monitor performance (success rate, confidence scores)
3. Collect failure cases
```

**Week 2: Iteration**
```
1. Analyze failure cases
2. Create prompt v2 with improvements:
   - Better instructions
   - More few-shot examples
   - Clearer output format
3. Deploy v2 alongside v1 (50/50 A/B test)
```

**Week 3: Evaluation**
```
1. Compare v1 vs v2 performance:
   - v1: 89% success rate, 0.85 avg confidence
   - v2: 94% success rate, 0.91 avg confidence
2. Promote v2 to 100% traffic
3. Deactivate v1
```

**Week 4: Continuous Improvement**
```
1. Monitor v2 performance
2. Identify new failure patterns
3. Create v3 with targeted improvements
4. Repeat cycle
```

### Benefits of Prompt Fine-Tuning Infrastructure

1. **No Code Deployments:** Update prompts via admin UI
2. **Data-Driven Optimization:** A/B test variants, promote winners
3. **Rapid Iteration:** Test new prompts in hours, not weeks
4. **Performance Tracking:** Know which prompts work best
5. **Domain Adaptation:** Create industry-specific prompts (tech, retail, finance)
6. **Failure Analysis:** Identify and fix extraction errors systematically

## Summary

### HyDE
- **What:** Generate hypothetical answer, use for retrieval
- **Why:** Improves semantic matching between queries and documents
- **When:** Questions that may not match document phrasing
- **Cost:** +1 LLM call per query

### Iterative Retrieval
- **What:** Multiple retrieval passes, refine based on gaps
- **Why:** Fills missing information, improves completeness
- **When:** Low-confidence initial results, multi-part questions
- **Cost:** +1-2 retrieval passes per query

### Prompt Fine-Tuning Infrastructure
- **What:** Database-driven prompt management with A/B testing
- **Why:** Continuous improvement without code changes
- **When:** Always (infrastructure for ongoing optimization)
- **Cost:** Minimal (database storage, admin UI development)

These three techniques work together to create a RAG system that:
1. **Retrieves better** (HyDE)
2. **Retrieves more completely** (Iterative Retrieval)
3. **Extracts better over time** (Prompt Fine-Tuning)

The result: **Institutional-grade accuracy** for competitive intelligence, MD&A insights, and footnote queries.
